# 长出来的工作流：脚手架、规范化与工具生长

[关于本项目](.claude/README.md)

## 引：方法不是设计出来的

在开始一项长跑之前，最常被问的问题是"先做什么"。答案常常落在工具、模板、约定上。但这些都属于实施层面。真正先要做的事反而朴素：把协作的规则写下来。这件事要生效的时间，比大多数人预期的要早。

本文的三件主题，来自一段持续了挺久的笔记与代码混合协作的工作流。它们不是被规划出来的，而是一百多次提交里自己长出来的。读起来像方法论，写起来像回顾录。

## 一、脚手架先于项目

任何长期由 AI 协作的工作区，在它还小的时候，就该有一份给 AI 读的人类规则，业内通常叫做"AI 行为契约"。这份契约不必一次写完，但必须不晚于第 N 次决策之前写出。所谓第 N 次决策，是指那些之后会被大量迁移路径依赖的具体判断：命名约定、提交颗粒度、不可逆操作的边界、归档的触发条件。

把目光放回这一段历史，那份契约文档在仓库第 6 次提交时就已经出现了，但到今天仍在被改写。它一开始就该是 v0，不是 v1。v1 一旦写定就会变成迁移成本——以后改它要付出的代价比改任何内容都重。

为什么必须先承认它是 v0？因为规则一旦被大型数据集的迁移路径依赖，再改就要做"规则迁移"，而规则迁移的代价比内容迁移大一个量级。常见的反模式是先写一份"完美版"的命名规范、提交规范、frontmatter 规范，等发现 edge case 时才发现整套都要重命名。那时候的重命名常常不得不拆成几十次细粒度的提交才安全。

v0 的正确姿态：先写一份能用的版本，把已知的不足有意地记在文末。这份"已知不足"清单本身就成了 v1 的候补，前提是它已经能让 v0 形态下的日常工作跑通。把 v0 当作"已完成"是错觉；当作"先跑起来，以后改"才是对的。

迁移路径是 v0 必须预留的，因为它一定会过时。预留靠两件事。一是命名前向兼容：旧名字段、空 fallback 解析、旧名做空指针等价处理。二是批量加扫描器兜底：单次迁移靠脚本，扫描器靠周期复检。脚本处理大量同质情况，扫描器负责捡漏扫尾。两者缺一不可。

仓库历史里有过不止一次"我以为的命名 v1"变成"明显 v0"的迁移。每次迁移之后都有一笔专门的补漏 commit，修扫描器未覆盖的角落。补漏 commit 本身不可怕，可怕的是把补漏当成创新。它们属于维护，不属于进展。

## 二、推进与规范化交替

工作流的心跳不是单向推进，是"推进一段、停下来规范化一段、再推进"。规范化不是"做完了顺手整理"，是显式阶段，要为它争取和推进同等的时间预算与可见度。

把规范化阶段显式化有几个理由。第一个，慢变量不会自己停。Broken wikilinks、tag 漂移、命名不一致——每一项都是慢的、累积的、单独看不严重的。合并起来是中后期的崩溃，而崩溃的修复成本远高于日常维护。第二个，显式触发便于回顾。"这周做了什么"在只有推进的仓库里几乎说不上来；在有规范化周期的仓库里清清楚楚，每一类改动属于哪一段、从 commit 历史里能独立检索。第三个，规范化阶段本身是双倍慢型：扫描一遍之后才能动手改，改完之后还要再扫一遍验证。这种节奏不适合"随项目顺手做"，必须独立成段才能被独立计时、独立回顾。

落到这段工作流上，提交历史里能找到十多条显式的"规范化 commit"——dates、tags、headers、template、plan naming，每一类都独立成批，每批内部也是细粒度的。把规范化摆成显式阶段，"我以为已经扫干净"这件事就可被证伪。

规范化最容易掉的坑是把它和"打扫卫生"画等号。打扫卫生只清可见脏污。规范化要清的是结构漂移——同一个对象在不同段落被以不同方式称呼。结构漂移只有显式扫描才能看见，因为它本身就是"看不见的问题"。

## 三、工具从痛点长出来

不要预先架构工具。工具的生命力来自它解决过一次真实的、重复的痛，不是来自一份抽象的"工具框架文档"。

3 次重复信号。一个工作步骤在不同的情境下被重复 3 次以上时，是时候把它沉成工具。重复的检测不需要阈值插件，只需要留意"上周我自己说过的某句话，又对一个新对象说了一遍"。

仓库里那些后来成为正式 skill 的工具，每一个都是先被手工累到"烦"才被立项的。最显眼的一个，从"给单篇笔记补 callout、mermaid、aliases、Dataview 查询"手动重复了几十次以后，长成了 vault-enhance skill；又一个，从"从混合 PDF 里抽真实图嵌入对应章节"手动重复十几次以后，长成了 pdf-gap-fill skill。工具追着痛点长，痛点追着情境冒。

工具的边界守则在"工具替代判断"处停下。判断归人类，资料收集归工具。工具若把判断也吞下，过程就藏进不透明的盒子里——这是协作型工作被自动化挤出去之后最常见的退化形态。

工具从诞生到成熟有一条反复出现的曲线：第一次是手工，第二次是手工，第三次还是手工，直到动手的人对这件重复开始厌烦，工具才正式出现。等不到那个时刻的工具体量小、被低估；超过那个时刻的工具来得太晚，带着一肚子没救的边角案例。两个偏离都很常见：前者是过早抽象，后者是拒绝抽象。判断"现在该出现"的信号只有一个——用户开始觉得烦。

## 收：演化路径，不是实施计划

把三件事串成一条链：

> **脚手架先于起步，规范化穿插于推进，工具生长于痛点之后。**

这不是一份可以照搬的实施计划，它是一条从长跑里观察出来的路径。能遵循它的工作流，几乎都不是一开始就立下的。它们是跑了一段之后，主事的人才看见"原来我早就在这么做"。

反过来读一遍：不要试图站在原地规划一条完美的协作路径。先把 v0 写出来，让规范化在长跑中自然出现的位置显形，让工具在痛点真的冒头时再沉下去。

让方法长出来。能被重写的方法，比不会过期的方法，更适合陪伴一段长跑。
