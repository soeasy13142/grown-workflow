# 命令：`/ecc:test-coverage` — 覆盖率分析

> 本命令基于 ECC（affaan-m/ECC, MIT）的 `/ecc:test-coverage`。
> 详见 [CREDITS.md](../../../CREDITS.md)。  ✅ v0 · 2026-07-10

## 何时用

- 功能开发完成后检查覆盖率
- 发现某模块测试不足时
- CI 流水线中覆盖率门禁失败时
- review 指出测试缺失后

## 真实示例

**场景：** TDD 实现了一个工具函数后检查覆盖率。

```
你： /ecc:test-coverage
    检查一下 .claude/scripts/utils.sh 的测试覆盖率。

ECC： 分析...
      文件: .claude/scripts/utils.sh
      行覆盖率: 72%
      分支覆盖率: 60%
      ❌ 未覆盖: slugify() 的空输入分支
      ❌ 未覆盖: validate_input() 的边界值
      ℹ️ 目标: ≥ 80%
```

**产出物：** 覆盖率报告，以及标记的未覆盖行/分支。

## 本仓库约定

1. **目标覆盖率 ≥ 80%** — 见 `.claude/rules/common/testing.md`
2. **不是数字游戏** — 不要仅为通过覆盖率而写无效断言。真实覆盖行为比数字重要
3. **核心模块优先** — 工具函数、数据层、API 路由覆盖优先级最高；UI 组件次之

## 边界与陷阱

| 情况 | 处理 |
|------|------|
| 覆盖率已达目标但测试质量差 | 检查测试是否有真实断言，还是只"跑过了" |
| 某些代码不可测试（配置项、常量） | 在覆盖率配置中排除 |
| 覆盖率工具未配置 | /ecc:test-coverage 会引导配置 |
| 本仓库无运行时代码 | 覆盖率分析不适用——跳过此命令 |

## 验证

跑完 `/ecc:test-coverage` 后检查：

- ✅ 覆盖率报告已生成并理解
- ✅ 未覆盖的边界已有计划补充
- ✅ 核心模块覆盖率 ≥ 80%

## 交叉引用

| 相关命令 | 关系 |
|---------|------|
| `/tdd` | 覆盖率不足时用 TDD 补充测试 |
| `/ecc:code-review` | 覆盖率分析结果可作 review 输入 |
| `/ecc:refactor-clean` | 低覆盖率区域可能是死代码候选 |

## 已知不足（v0）

- 本仓库当前以文档为主，覆盖率分析的实际价值有限
- 需要配合具体的测试框架（jest / bats / pytest），取决于当前项目语言
- ECC 的覆盖率分析能力取决于底层工具链支持
