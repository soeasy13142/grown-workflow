# 命令：`/e2e` — 端到端测试

> 本命令基于 ECC（affaan-m/ECC, MIT）的 `/e2e`。
> 详见 [CREDITS.md](../../../CREDITS.md)。  ✅ v0 · 2026-07-10

## 何时用

- 完整的用户流程验证
- 跨模块/跨服务的集成场景
- 发布前的回归检查
- 验证多个 ECC 命令的串链执行结果

## 真实示例

**场景：** 验证一个技能从 plan → 开发 → review 的完整工作流。

```
你： /e2e
    验证一下 ecc-workflow 的使用流程：
    1. 用 /ecc:plan 规划一个新 skill
    2. 按 plan 实施
    3. 用 /ecc:code-review 审查

ECC： 分析...
      模拟流程：
      ✅ Phase 1: plan 生成 → docs/plans/ 下有文件
      ✅ Phase 2: 文件结构符合 plan
      ⚠️ Phase 3: review 通过——但项目缺少测试
      ℹ️ 建议补 /ecc:test-coverage
```

## 本仓库约定

1. **先单元测试（/tdd），再 e2e** — e2e 覆盖关键路径，不试图覆盖所有分支
2. **e2e 应可重复运行** — 不依赖外部状态
3. **失败时提供足够上下文** — 截图、日志、请求/响应快照（如适用）

## 边界与陷阱

| 情况 | 处理 |
|------|------|
| 项目尚无 e2e 基础设施 | /e2e 会生成测试框架配置，首次运行需验证 |
| e2e 测试依赖外部服务 | 用 mock 或 docker 替代，确保可离线运行 |
| e2e 测试运行时间过长 | 拆成多个独立的 e2e 用例，分批执行 |
| 本仓库是纯文档项目 | e2e 的实际价值有限——考虑是否真的需要 |

## 验证

跑完 `/e2e` 后检查：

- ✅ e2e 用例覆盖了关键用户路径
- ✅ 测试可重复运行（无状态泄露）
- ✅ 失败测试有清晰的错误信息

## 交叉引用

| 相关命令 | 关系 |
|---------|------|
| `/tdd` | e2e 之前应先有单元测试 |
| `/ecc:test-coverage` | e2e 之后检查整体覆盖率 |
| `/ecc:code-review` | 发布前 review |

## 已知不足（v0）

- 本仓库是文档/配置项目，e2e 的实际适用场景有限
- /e2e 命令主要服务于包含运行时代码的子项目
- 需要明确何时才值得为这个仓库写 e2e
